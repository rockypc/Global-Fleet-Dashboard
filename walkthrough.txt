# Global Fleet Dashboard — Application Index & Architecture

## Overview

The **Global Fleet Dashboard** is a **single-file, client-side aviation intelligence platform** built entirely within one 5,800-line `index.html` file (~359 KB). It serves as a centralized command center for **asset remarketing, RFP matching, fleet analytics, transition risk management, and competitive intelligence** across the global commercial aviation fleet.

> [!IMPORTANT]
> The entire application (HTML structure, CSS, and JavaScript) lives in **one monolithic `index.html`** — there is no build system, no framework, and no backend server. Everything runs in the browser.

## Data Sources

### Primary: `Aircraft Detail.csv` (~35 MB)
- Fetched **live from GitHub** at startup: `https://raw.githubusercontent.com/rockypc/Global-Fleet-Dashboard/refs/heads/main/Aircraft%20Detail.csv`
- Contains the complete global fleet database (Airbus & Boeing only, filtered at load)
- Key fields: Serial Number, Operator, Type, Status, Lease End, Age, Market Value, Engine specs, Seating, MTOW, etc.

### Supplementary: `Consolidated Scores Sheet.csv` (~16 KB)
- **Airline credit ratings** — 196 airlines with Ishka Credit Scores (e.g., `BBB-`, `CCC`, `A-`)
- Joined to the main dataset by operator name at load time
- Contains: Business Model, Region, Financial/Traffic/Yield indicators, Credit Score, Outlook

### Data Loading Waterfall
```mermaid
flowchart TD
    A[App Loads] --> B{Fetch from GitHub}
    B -->|Success| C[Parse CSV + Filter Airbus/Boeing]
    B -->|Fail| D{Check localforage cache}
    C --> E[Cache to IndexedDB]
    E --> F[Initialize Dashboard]
    D -->|Cache exists| G[Load cached data]
    D -->|No cache| H[Show manual upload UI]
    G --> F
    H -->|User uploads CSV| I[Parse + cache + init]
```

## Data Processing Pipeline

The `initDash()` function runs a **5-step pipeline** on every data load:

```mermaid
flowchart LR
    S1["Step 1<br/>Parse & Normalize"] --> S2["Step 2<br/>Operator Rollups"]
    S2 --> S3["Step 3<br/>Top-25 Hit List"]
    S3 --> S4["Step 4<br/>Score Pass 2 + TAM"]
    S4 --> S5["Step 5<br/>Dynamic Hooks"]
```

### Step 1: Parse, Validate, Normalize
- Lease End dates → robust parser (handles `MMM YYYY`, ISO, US formats)
- Age → cascading: `Age` field → `Delivery Date` → `Build Year`
- Market Value → strips `$`, commas, auto-detects scale ($M vs raw USD)
- Status → normalized to: In Service, Storage, On Order, Other
- Derived fields: age buckets, TATS scoring components, MRO risk, USM prime candidates, retirement probability score

### Step 2: Operator Rollups & Tier Classification
- Groups by operator, counts active/stored fleet
- Top 10 operators by fleet size → **Tier 1** classification

### Step 3: Top-25 Hit List
- Filters to lease ends within `2025-11-30` to `2026-12-31`
- Score Pass 1: storage bonus, age bonus, Tier 1 bonus, estimated lease penalty
- Sorts by score → market value → days to lease end

### Step 4: Opportunity Score + TAM
- Full **TATS Opportunity Score** (0–100) combining:
  - Days to lease end (30pts max)
  - Storage status (20pts)
  - Age tiers (25pts max)
  - Operator tier (15pts)
  - Top-25 membership (10pts)
  - Estimated lease penalty (-15pts)
- Risk tiers: ≥70 = High Risk, ≥40 = Medium Risk, else Low Risk
- Estimated TATS USD = market value × 0.002

### Step 5: Dynamic Hook Generation
- Auto-generates personalized outreach text per aircraft for the TATS Prospecting pipeline

## Architecture Observations

> [!NOTE]
> **Strengths**: Zero build complexity, fully offline-capable via localforage caching, GitHub-based live sync, comprehensive aviation domain coverage, print-to-PDF for IC tear sheets.

> [!WARNING]
> **Risks**: The single-file monolith at 5,800+ lines is approaching maintainability limits. Adding features like MRO & USM (`renderUSM`), Exposure (`renderExposure`), and History (`renderHistorical`) are referenced but appear to be stubs (`typeof renderUSM === "function"` guards). Splitting into modules or adopting a build system would improve development velocity.
